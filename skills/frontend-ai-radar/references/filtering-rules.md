# Filtering Rules

Use these rules when turning noisy public sources into a frontend engineering radar.

## Keep

- Original engineering opinions from builders, framework maintainers, infra teams, and senior frontend practitioners.
- Concrete changes in frontend architecture: rendering strategy, state/data flow, routing, SSR/RSC, micro-frontends, monorepos, package boundaries, design systems, and component governance.
- Engineering-management signals: code ownership, review loops, release safety, CI, test strategy, observability, rollout practice, dependency governance, and build-cost reduction.
- Tooling and DX: Vite, Rspack, Turbopack, Webpack, Rollup, esbuild, SWC, TypeScript, linting, formatting, Storybook, Playwright, Vitest/Jest, package managers, and repo automation.
- UI platform depth: performance, accessibility, CSS architecture, animation, WebGL, Three.js, WebGPU, canvas, visualization, and 3D scene interaction.
- AI-assisted development patterns: Codex/Claude Code workflows, prompt design, evals, agent permissions, test harnesses, local knowledge bases, issue-to-PR automation, and review automation.

## Skip

- Generic AI hype with no technical or workflow detail.
- Reposts, quote posts, and "great event" posts without commentary.
- Course ads, conference CFPs, hiring posts, and newsletter promotions unless they contain concrete technical takeaways.
- Pure product announcements with no architecture, workflow, or implementation implications.
- Content with no direct source link.

## Ranking Heuristic

Rank higher when an item answers at least one of these:

- What should a frontend architecture lead change or watch?
- What could improve the team's engineering process?
- What can be turned into a Codex agent workflow, prompt, test, or checklist?
- What affects performance, reliability, build speed, or maintainability?
- What matters for complex UI/3D delivery rather than generic web pages?

When in doubt, cut the item and keep the digest dense.
