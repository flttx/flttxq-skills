# 测试分层判定（Unit vs Integration）

## 1. 判定优先级
1. 先看目标：验证局部逻辑还是跨模块协作。
2. 再看边界：是否跨越组件-状态-服务等边界。
3. 最后看依赖：是否需要真实组装多个模块。

## 2. 单元测试（Unit）判定清单
- 被测对象可独立实例化（函数、hook、store action、纯组件行为）。
- 外部依赖均可替换为 mock/stub。
- 断言聚焦输入-输出、状态变化、调用参数。
- 不需要真实路由跳转、不需要真实网络链路。

常见对象：
- 工具函数（URL 构建、格式化、校验）
- 服务封装函数（参数拼装、头部注入）
- Store 单个 action
- 组件局部交互（按钮点击导致回调触发）

## 3. 集成测试（Integration）判定清单
- 被测行为依赖多个模块协作（组件 + store + service）。
- 需要验证“接线是否正确”，而不仅是单点函数结果。
- 保留真实内部组合，只 mock 外部边界（网络、原生 API、第三方 SDK）。
- 断言应覆盖一条用户行为链路的结果。

常见对象：
- 页面路由组件把 props/path 正确传给通用容器
- 登录/注销链路的状态切换与缓存清理协作
- 录音/导入页面在授权、弹窗、请求之间的协作

## 4. 反模式
- 用单测名义写超大场景（断言 20+ 细节、mock 全世界）。
- 集成测试里过度 mock，导致只剩“伪单测”。
- 断言 UI 树内部细枝末节，改样式即失败。

## 5. 命名与命令建议
- Unit：`*.test.ts(x)`
- Integration：`*.integration.test.ts(x)`
- 分层命令：
  - `test:unit` 忽略 `*.integration.test.*`
  - `test:integration` 只跑 `*.integration.test.*`
